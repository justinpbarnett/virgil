# Feature: Todo Pipe

## Metadata

type: `feat`
task_id: `todo-pipe`
prompt: `Add a todo pipe backed by SQLite that supports add, list, done, remove, reorder, and edit actions. Writes memory nodes on create/complete for context assembly. Includes a pipe_affinity schema field for future autonomous execution.`

## Feature Description

Virgil needs a personal todo list. The `todo` pipe is a deterministic pipe that manages structured todo items in SQLite via the existing store. It supports full CRUD plus priority reordering and completion tracking.

Todo items are the source of truth in a dedicated `todos` table, but lifecycle events (creation, completion) also write memory nodes into the existing `memories` table so that context assembly can answer questions like "what did I get done this week?" without the chat pipe needing to know about the todo schema.

Each todo item carries a `pipe_affinity` text field (comma-separated pipe names) that records which pipes would be needed to execute the item. This field is not exposed as a flag yet — it exists in the schema for future use when Virgil can scan the todo list and autonomously act on items whose required pipes are available.

## Implementation Plan

### Phase 1: Schema

Create the `todos` table via a goose migration. Add CRUD methods to the store package.

### Phase 2: Pipe Implementation

Build the `todo` pipe handler following the calendar pipe pattern — action flag dispatches to per-action handlers. Each handler calls store methods and returns an envelope.

### Phase 3: Memory Integration

On `add` and `done` actions, write a memory node (kind `invocation`, source_pipe `todo`) into the `memories` table. On `done`, also create a `refined_from` edge linking the completion memory to the creation memory, so the graph captures task lifecycle.

### Phase 4: Routing

Define vocabulary, triggers, and format templates in `pipe.yaml` so the router can classify todo-related signals.

## Schema

### `todos` table

```sql
CREATE TABLE IF NOT EXISTS todos (
    id TEXT PRIMARY KEY,
    title TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'pending',
    priority INTEGER NOT NULL DEFAULT 3,
    due_date TEXT,
    tags TEXT NOT NULL DEFAULT '',
    pipe_affinity TEXT NOT NULL DEFAULT '',
    memory_id TEXT,
    created_at INTEGER NOT NULL,
    completed_at INTEGER
);

CREATE INDEX IF NOT EXISTS idx_todos_status ON todos(status);
CREATE INDEX IF NOT EXISTS idx_todos_priority ON todos(priority);
CREATE INDEX IF NOT EXISTS idx_todos_status_priority ON todos(status, priority);
```

**Field notes:**
- `id` — UUID, generated by `store.newID()`.
- `status` — `pending` or `done`.
- `priority` — 1 (highest) to 5 (lowest). Default 3.
- `due_date` — ISO 8601 date string (`2026-03-10`) or empty. Stored as TEXT for simplicity since there's no date arithmetic needed in SQL.
- `tags` — Comma-separated, same convention as `memories.tags`.
- `pipe_affinity` — Comma-separated pipe names (e.g., `calendar,phone,draft`). Not user-facing yet. Populated in the future by the AI planner when items are created.
- `memory_id` — The `memories.id` of the invocation memory node written when this todo was created. Used to create graph edges on completion.
- `created_at` — Unix nanoseconds, same convention as `memories.created_at`.
- `completed_at` — Unix nanoseconds, set when status changes to `done`. NULL while pending.

### Store Methods

```go
// Todo represents a todo list item.
type Todo struct {
    ID          string    `json:"id"`
    Title       string    `json:"title"`
    Status      string    `json:"status"`
    Priority    int       `json:"priority"`
    DueDate     string    `json:"due_date,omitempty"`
    Tags        []string  `json:"tags,omitempty"`
    CreatedAt   time.Time `json:"created_at"`
    CompletedAt time.Time `json:"completed_at,omitempty"`
}

func (s *Store) AddTodo(title string, priority int, dueDate string, tags []string) (Todo, error)
func (s *Store) ListTodos(status string, limit int) ([]Todo, error)
func (s *Store) GetTodo(id string) (Todo, error)
func (s *Store) UpdateTodo(id string, updates map[string]string) error
func (s *Store) CompleteTodo(id string) error
func (s *Store) DeleteTodo(id string) error
func (s *Store) ReorderTodo(id string, newPriority int) error
func (s *Store) SetTodoMemoryID(id, memoryID string) error
```

`ListTodos` orders by `priority ASC, created_at ASC` (highest priority first, then oldest first within same priority). When `status` is empty, returns all items. Default limit is 25.

## Pipe Definition

### `pipe.yaml`

```yaml
name: todo
description: Manages a personal todo list — add, complete, edit, remove, reorder, and list tasks.
category: general

triggers:
  exact:
    - "show my todos"
    - "what's on my todo list"
    - "what do I need to do"
  keywords:
    - todo
    - task
    - todos
    - tasks
    - checklist
    - backlog
  patterns:
    - "add {topic} to my todo list"
    - "add {topic} to my todos"
    - "mark {topic} as done"
    - "check off {topic}"
    - "remove {topic} from my todos"
    - "prioritize {topic}"
    - "reorder {topic}"

flags:
  action:
    description: Which todo operation to perform.
    values: [add, list, done, remove, edit, reorder]
    default: list
  title:
    description: The todo item title (for add action).
    default: ""
  id:
    description: Todo item ID (for done, remove, edit, reorder actions).
    default: ""
  priority:
    description: Priority level 1 (highest) to 5 (lowest).
    values: ["1", "2", "3", "4", "5"]
    default: "3"
  due:
    description: Due date (ISO 8601 date or natural language like "friday").
    default: ""
  tags:
    description: Comma-separated tags.
    default: ""
  status:
    description: Filter by status (for list action).
    values: [pending, done, all]
    default: pending
  limit:
    description: Maximum items to return (for list action).
    default: "25"

format:
  list: |-
    {{if eq .Count 0}}No todos.{{else}}{{range .Items}}
    {{if eq .status "done"}}[x]{{else}}[ ]{{end}} {{.title}}{{if .due_date}} (due {{.due_date}}){{end}} [p{{.priority}}] {{.id}}{{end}}{{end}}
  structured: |-
    {{if .status}}{{if eq .status "done"}}Completed{{else}}Added{{end}}: {{.title}}{{end}}{{if .priority}} [p{{.priority}}]{{end}}

memory:
  disabled: true

vocabulary:
  verbs:
    todo: todo
    task: todo
  types: {}
  sources:
    todos: todo
    tasks: todo
  modifiers: {}

templates:
  priority: 50
  entries:
    - requires: [verb]
      plan:
        - pipe: todo
```

## Handler Design

The handler follows the calendar pipe pattern — a top-level `NewHandler` returns a `pipe.Handler` closure that dispatches on the `action` flag.

### Action: `add`

1. Read `title` from flags. If empty, try to extract from envelope content.
2. Parse `priority`, `due`, `tags` from flags.
3. Call `store.AddTodo(title, priority, due, tags)`.
4. Write an invocation memory: `store.SaveInvocation("todo", signal, "Added todo: "+title)` — returns a memory ID.
5. Link the memory to the todo: `store.SetTodoMemoryID(todo.ID, memoryID)`.
6. Return structured envelope with the created todo.

### Action: `list`

1. Read `status` and `limit` from flags.
2. Call `store.ListTodos(status, limit)`.
3. Return list envelope.

### Action: `done`

1. Read `id` from flags. If empty, try fuzzy title match from envelope content.
2. Call `store.GetTodo(id)` to get the todo (need `memory_id` for graph edge).
3. Call `store.CompleteTodo(id)`.
4. Write a completion memory: `store.SaveInvocation("todo", signal, "Completed todo: "+todo.Title)` — returns a new memory ID.
5. If the todo has a `memory_id`, create a `refined_from` edge from the completion memory to the creation memory via `store.CreateEdge`.
6. Return structured envelope confirming completion.

### Action: `remove`

1. Read `id` from flags.
2. Call `store.DeleteTodo(id)`.
3. Return structured envelope confirming deletion.

### Action: `edit`

1. Read `id` from flags.
2. Collect updates from flags (`title`, `priority`, `due`, `tags`).
3. Call `store.UpdateTodo(id, updates)`.
4. Return structured envelope confirming update.

### Action: `reorder`

1. Read `id` and `priority` from flags.
2. Call `store.ReorderTodo(id, priority)`.
3. Return structured envelope confirming new priority.

### Fuzzy Title Matching for `done`

When `id` is empty but the envelope content or signal contains text (e.g., "mark 'buy groceries' as done"), the handler queries `ListTodos("pending", 100)` and finds the first item whose title contains the search text (case-insensitive). This is a convenience — the canonical path is by ID.

## Subprocess Entry Point

```go
// internal/pipes/todo/cmd/main.go
package main

import (
    "os"

    "github.com/justinpbarnett/virgil/internal/pipehost"
    "github.com/justinpbarnett/virgil/internal/pipes/todo"
    "github.com/justinpbarnett/virgil/internal/store"
)

func main() {
    logger := pipehost.NewPipeLogger("todo")

    s, err := store.Open(os.Getenv(pipehost.EnvDBPath))
    if err != nil {
        pipehost.Fatal("todo", err.Error())
    }
    defer s.Close()

    logger.Info("initialized")
    pipehost.Run(todo.NewHandler(s, logger), nil)
}
```

## Testing Strategy

### Unit Tests

- **Store tests** (`internal/store/todo_test.go`): Test each CRUD method against a real SQLite database (temp file). Verify ordering, filtering, timestamp handling, tag serialization.
- **Handler tests** (`internal/pipes/todo/todo_test.go`): Test each action with a real store. Verify envelope fields (pipe, action, content_type, error). Verify memory node creation on add/done. Verify graph edge creation on done.

### Edge Cases

- Add with empty title → fatal error.
- Done/remove/edit with non-existent ID → fatal error with clear message.
- Done by fuzzy match when multiple items match → picks first by priority.
- Done by fuzzy match with no matches → fatal error.
- List with `status=all` → returns both pending and done items.
- Priority out of range (< 1 or > 5) → clamp to 1-5.
- Duplicate titles → allowed (IDs are unique).
- Tags with commas in values → not supported (comma is delimiter). Document this.

## Risk Assessment

- **Low risk.** This is a new pipe with a new table — no existing functionality is modified.
- **Migration safety.** The goose migration is additive (new table only). The Down migration drops the table cleanly.
- **Store concurrency.** The store uses `MaxOpenConns(1)` and WAL mode, so concurrent access from the todo pipe subprocess is safe.
- **Memory node volume.** Each add/done writes one memory node. This is the same rate as any other pipe invocation and well within normal usage patterns.

## Open Questions (Unresolved)

1. **~~Should the calendar pipe's `todo` keyword be removed?~~** Yes. Remove `todo` from the calendar pipe's keywords — todo items are not calendar events. The AI planner can compose `todo → calendar` pipelines when a todo item involves scheduling. **Decision: Remove it as part of this feature.**

2. **Should `done` support batch completion?** E.g., "mark all shopping todos as done." **Recommendation:** Not in v1. Single-item completion is sufficient. Batch operations can be added later with a `tag` filter flag.

3. **Natural language date parsing for `due` flag.** Should the todo pipe parse "friday" → "2026-03-06"? **Recommendation:** Yes, reuse the `when` parser already imported by the calendar pipe. Parse at the handler level before storing. Store the resolved ISO date.
